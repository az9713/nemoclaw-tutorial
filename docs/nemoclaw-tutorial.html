NemoClaw Tutorial

Architecture, communication flows, security model, and how OpenClaw, OpenShell, Docker, and NemoClaw work together.

What is NemoClaw?

NemoClaw is an NVIDIA open-source project that runs OpenClaw (an autonomous AI agent) safely inside OpenShell (a secure runtime) with inference routed through NVIDIA's cloud. It automates the entire process of sandboxing an AI agent so that every network request, file access, and model call is governed by declarative policy.

The One-Sentence Summary

NemoClaw is an OpenClaw plugin that automates putting OpenClaw inside an OpenShell sandbox, built on Docker containers, with NVIDIA inference and declarative security policies.

Why Docker if We Have OpenShell?

Key Insight

OpenShell doesn't replace Docker — it's built on top of it. Docker is the foundation; OpenShell is a security layer added on top. Without Docker, OpenShell has nothing to build on.

Every openshell command translates to Docker commands internally:

You RunOpenShell Internally Does
openshell gateway startdocker run (starts k3s container)
openshell sandbox createdocker build + docker run
openshell sandbox connectdocker exec
openshell sandbox deletedocker stop + docker rm

What each layer adds:

CapabilityDocker AloneDocker + OpenShell
Isolated filesystem✓ container has its own FS✓ + Landlock per-path read/write rules
Isolated network✓ own network namespace✓ + YAML policies per host/port/method
Isolated processes✓ own PID namespace✓ + seccomp syscall filtering
API credentialsPass as env vars (visible)Injected at proxy (invisible to app)
Monitoringdocker logsReal-time TUI with approval queue
Inference routingManual configurationTransparent proxy intercepts & routes

Analogy: Docker gives you a room. OpenShell gives you a room with locks, cameras, and an alarm system. NemoClaw is the contractor who builds the room, installs the locks, and puts the AI robot inside.

Setup Flow: nemoclaw onboard

  1. NemoClaw→Docker Check Docker is running (docker info)
  2. NemoClaw Check OpenShell is installed (openshell --version); install if missing
  3. NemoClaw→Docker Check cgroup v2 config (/etc/docker/daemon.json)
  4. NemoClaw Detect GPU hardware (nvidia-smi or system_profiler)
  5. NemoClaw→OpenShell→Docker Start gateway: openshell gateway start → Docker creates k3s container
  6. NemoClaw→Docker Stage build context: copy Dockerfile + plugin + blueprint + scripts
  7. NemoClaw→OpenShell→Docker Create sandbox: openshell sandbox create --from Dockerfile → Docker builds image + starts container with Landlock/seccomp/netns
  8. NemoClaw Prompt for NVIDIA API key → save to ~/.nemoclaw/credentials.json (mode 0600)
  9. NemoClaw→OpenShell Create provider: openshell provider create --credential ...
  10. NemoClaw→OpenShell Set inference route: openshell inference set --provider nvidia-nim
  11. OpenClaw OpenClaw starts inside sandbox with NemoClaw plugin pre-installed
  12. NemoClaw→OpenShell Apply policy presets: openshell policy set for pypi, npm, etc.
  13. NemoClaw→OpenShell→Docker Port forward: openshell forward start --background 18789

Defense-in-Depth Security Model

NemoClaw uses five independent security layers. Even if one layer is compromised, the others still protect:

Layer 5: Inference Routing

All model calls routed through OpenShell gateway. Agent never sees API keys. Configurable per-provider, hot-reloadable.

Layer 4: Network Policy

Declarative YAML allowlist. Per-endpoint host/port/method/path rules. Per-binary restrictions. Hot-reloadable. Source: openclaw-sandbox.yaml

Layer 3: Process Isolation

Runs as unprivileged sandbox user. seccomp syscall filtering blocks dangerous operations. No root capabilities.

Layer 2: Filesystem Isolation (Landlock)

Read-only: /usr, /lib, /proc, /app, /etc. Read-write: /sandbox, /tmp. All other paths denied.

Layer 1: Container Isolation (Docker)

Network namespace, PID namespace, UID/GID restrictions. Provides the baseline isolation that all other layers build on.

Network Policies

The base policy in openclaw-sandbox.yaml declares which endpoints the sandbox can reach. Each policy entry specifies host, port, allowed HTTP methods, and which binaries can use it:

network_policies:
  nvidia:
    endpoints:
      - host: integrate.api.nvidia.com
        port: 443
        rules:
          - allow: { method: "*", path: "/**" }
    binaries:
      - { path: /usr/local/bin/claude }
      - { path: /usr/local/bin/openclaw }

  github:
    endpoints:
      - host: github.com
        port: 443
        access: full
    binaries:
      - { path: /usr/bin/gh }
      - { path: /usr/bin/git }

Credential Injection

User provides API key
     │
     ▼
Stored in ~/.nemoclaw/credentials.json (mode 0600)
     │
     ▼
Passed to openshell provider create --credential "NVIDIA_API_KEY=nvapi-..."
     │
     ▼
OpenShell stores credential in its own secure config
     │
     ▼
When sandbox makes inference request:
  1. Request goes to https://inference.local/v1  (inside sandbox)
  2. OpenShell proxy intercepts
  3. Proxy injects Authorization: Bearer <key>
  4. Proxy forwards to real endpoint
  5. Response returns to sandbox

The sandbox NEVER sees the actual API key.

FAQ

Can I use OpenClaw without NemoClaw?

Yes. OpenClaw works standalone. But it runs unsandboxed with full host access.

Can I use OpenShell without Docker?

No. OpenShell's gateway is a Docker container running k3s. Sandboxes are Docker containers. Docker is a hard requirement.

Why does NemoClaw talk to Docker both directly AND through OpenShell?

Through OpenShell: for sandboxes (need security layers). Directly: for NIM containers (trusted services that don't need sandboxing).

How many Docker containers run after setup?

2 (cloud inference): OpenShell gateway + sandbox. 3 (local NIM): + NIM model server.

Where does NemoClaw end and OpenShell begin?

NemoClaw builds CLI arguments and calls openshell. Everything before the openshell command is NemoClaw's logic. Everything after is OpenShell's.

Which project handles security at runtime?

OpenShell enforces (Landlock, seccomp, proxy). NemoClaw configured it at setup time. OpenClaw is the thing being secured.

What happens if the agent tries to reach an unlisted host?

OpenShell blocks the request and surfaces it in the TUI (openshell term). The operator can approve once, add permanently, or deny.
